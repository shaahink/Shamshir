## 5. What is thin, weak, or assumed (honest assessment)

### Shallow implementations

1. **EntryDiagnosis.ComputeEma double-counts seed values.** The EMA seed is computed as SMA of the last `period` values, then the EMA iteration re-walks those same values. This converges toward the most recent values but is not standard EMA. For a diagnostic tool comparing relative feature values, the impact is small. Not fixed — would change the 10 unit tests.

2. **EntryDiagnosis.ComputeSqueezeAge has off-by-one loop and inconsistent BB width.** The loop iterates `i <= n` (one extra iteration, guarded by Min), and the BB width uses `2*std/sma` while the comment says `4*std/sma`. Both issues are scale-invariant for the purpose of percentile comparison. Not fixed — fixing would require re-validating all squeeze-age tests.

3. **ExitLabController exitlab-eval playbook steps require non-empty positionIds.** The `regime-calibration.json` playbook ships with empty `positionIds: []`. The controller rejects empty positionIds. The playbook step runner now resolves `$exploreRun` references correctly but the API still requires explicit positionIds. The playbook needs a pre-step to auto-discover positions from `GET /api/exit-lab/runs/{runId}/excursions`. **NOT FIXED** — requires API changes.

### Known deferred issues (open traps from AGENTS.md, not regressions)

4. **Open Trap #1: Session labels not wired into TradeExcursions.** SessionDetector exists but trade excursions don't carry session labels. The regime calibration can label entry bars but not the full excursion.

5. **Open Trap #2: SpreadVolNoTradeFilter no strategy config wiring.** The filter class exists and works but isn't wired into the strategy configuration system. No strategy currently activates it.

6. **Open Trap #3: Playbook 3 (triage-sweep.json) not created.** Referenced in P3.5 plan but never shipped.

7. **Open Trap #4: BlockBootstrapper writes bars to real MarketDataShard.** Bootstrap bars are written to `SqliteMarketDataStore` alongside real market data. No isolation.

8. **Open Trap #5: Fixed in this session** (see §3).

9. **Open Trap #6: EntityAuditableTests red on ExitCalibrationEntity.** Pre-existing, unrelated to P6.

10. **Open Trap #7: Any session touching web-ui/src/*.ts MUST run `npm run build`.** Not triggered by this audit.

### Known design limitations

11. **BlockBootstrapController constructs synthetic runs that don't have weekend gaps.** `RemapTimestamps` creates consecutive sequential bars. The engine sees continuous trading with no pauses. This changes behavior subtly vs real data (e.g., no weekend rollovers, different bar-count-based indicators).

12. **EntryQualityController makes one DB round-trip per trade for median ATR** (`GetMedianAtrPipsAsync`). For 100 trades this is 100 extra queries. Caching per (symbol, tf) would be trivial.

13. **ExitLabController.SaveCalibration uses DELETE + INSERT for upsert.** If `SaveChangesAsync` fails mid-way, data is lost. Should use a transaction.

14. **MetaAllocator.Summarize throws on empty input** (no guard clause). Callers always pass non-empty lists, but the public API doesn't protect.

15. **BlockBootstrapper.PartitionIntoBlocks has `current!.Add(bar)` null-forgiving.** Safe at runtime (current is always set on first iteration) but the null annotation is misleading.

## 7. Risks the next phase should watch

1. **The playbook ecosystem is structurally complete but not end-to-end tested.** The HttpStepRunner has 15 step kinds, ResolveParams for context references, and ResolveRef for named runId lookups — but no integration test ever runs a full playbook against a live API. The playbooks parse (10/10 ShippedPlaybook_Parses) but their actual HTTP contracts have never been smoke-tested.

2. **Block-bootstrap controller writes to real MarketDataShard.** If the operator runs a bootstrap with write access, synthetic bars pollute the production market data store. This needs a sandbox data source or a writable dedicated shard.

3. **Entry diagnosis OLS uses hand-rolled matrix inversion.** Gauss-Jordan with a 1e-14 singularity threshold. The algorithm is proven correct by tests with known coefficients, but numerical stability on real-world floating-point bar data is untested. No Cholesky/QR fallback.

4. **Spread/vol filter is not wired into any strategy config.** The class exists but is inert — no strategy activates it. The `spread-vol-filter.json` playbook starts two identical runs (both without the filter). The A/B measurement is conceptual only.

5. **The regime playbook's exitlab-eval steps send empty positionIds** and require manual population of position IDs before the playbook can actually run.
