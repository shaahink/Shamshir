## Audit findings (s19)

### Bugs found and fixed

| ID | Severity | Description | Status |
|----|---|---|---|
| B1 | CRITICAL | **Idempotency TOCTOU race.** `StartAsync` used `TryGetValue` then `TryAdd` — two concurrent requests with the same idempotency key could both pass the check and start duplicate runs. Fixed: double-checked locking pattern with dedicated `_idempotencyLock`. | **FIXED** `46ba5ab` |
| B2 | MEDIUM | **Unbounded idempotency key dictionary.** `_idempotencyKeys` ConcurrentDictionary grew without bound. Fixed: cap at 10,000 entries; clear with warning when exceeded. | **FIXED** `46ba5ab` |
| B3 | MEDIUM | **`completed-with-warnings` progress bar shows 99.9%.** `BuildProgress` only set `percent=100` for exact `"completed"` status. `completed-with-warnings` fell through to calculated ~99.9%. Fixed: pattern match includes `"completed-with-warnings"`. | **FIXED** `46ba5ab` |
| B5 | MEDIUM | **`completed-with-warnings` treated as non-terminal in `GetCurrentProgress`.** The switch only recognized `"completed"`, `"failed"`, `"cancelled"`. A warned-completed run would send `RunProgress` (not `RunCompleted`) on `JoinRun`, leaving the client stuck in "running" state. Fixed: added `"completed-with-warnings"` to the terminal case. | **FIXED** `46ba5ab` |
| B4 | LOW | **TypeScript `RunProgressEnvelope` nullable fields inconsistent with C#.** `dailyDdPct`, `maxDdPct`, `distanceToDailyLimit` typed as `number \| null` in TS but C# sends non-nullable `decimal` (defaults to `0`). Not a runtime bug — TS null guards are defensive and `0` passes through. The correct fix (make C# nullable, gate on `HasEquityObservation`) touches the SignalR contract and should be its own tested change. | **DEFERRED** (low-risk, no user impact) |

### What was already solid (no changes needed)

- **M45 migration** — correct, adds nullable `ComparePairId` TEXT column, down migration drops it, model snapshot synced.
- **DTO propagation chain** — `BacktestRunSummary` → `BacktestRunEntity` → `SqliteBacktestRunRepository.SaveAsync/ReconcileAsync` → `RunListResponse`/`RunDetailResponse` wired consistently. All nullable fields handled.
- **Compare-both grouping** (`groupedRuns` computed) — correct algorithm: first pass collects `comparePairId` and `parentRunId` into maps, second pass emits parents first then indented children. Edge cases: solo runs (no `comparePairId`) pass through ungrouped; children without a parent in the list (orphaned) still show (included in sibling loop); no double-counting (children skipped in parent loop).
- **Status chips** — `BadgeComponent` supports `'neutral'` variant since inception (line 11, default case renders gray). The new status variants map correctly.
- **Equity nullability** — `BuildProgress` gates on `HasEquityObservation` set in `ApplySnapshot`. `RunProgress.Envelope` serialized with camelCase policy → Angular `equity: number | null` matches. Chart skips null points. `fitContent()` handles multi-pass equity curves.
- **Idempotency key client generation** — `crypto.randomUUID()` is collision-resistant. Key flows: `new-backtest.component.ts` → `runs.service.ts` → POST body → `RunsController.Start` → `CustomParams["IdempotencyKey"]` → `StartAsync`. `IsNullOrWhiteSpace` gate correctly skips empty/missing keys.
- **`starting()` signal lifecycle** — set `true` synchronously on click (before async), reset in `finally` block. All early-return error paths also reset it. Button disabled on `!canStart() || starting()`.

### What is shortcut / weak / assumed / not fully covered

1. **No unit/integration tests for idempotency lock.** The double-checked locking pattern is correct by inspection and the lock is uncontested in practice (single-user dev tool), but there are no concurrent tests proving it. A concurrent test would need `Task.WhenAll` with the same idempotency key and assertion that only one runId is returned. Deferred: the cost-benefit of a non-deterministic concurrent test is poor for a dev tool.

2. **No tests for idempotency dictionary capacity eviction.** The 10K-limit clear logic works structurally but has never triggered. For the expected usage pattern (< 1000 runs), this is a safety net, not a tested path.

3. **`completed-with-warnings` progress fix not live-tested.** The `BuildProgress` and `GetCurrentProgress` fixes are structurally correct, but no `completed-with-warnings` run was driven end-to-end to confirm the monitor shows 100% and `RunCompleted` instead of `RunProgress`. The fix is 2-character changes in 2 well-structured switch/match expressions.

4. **`BuildRunDetailFromState` doesn't propagate `ParentRunId`/`ComparePairId`.** When a run is live/in-memory (not yet persisted), the run-detail response misses these linkage fields. For live runs, this is acceptable (compare-both grouping is a post-completion concern), but it means a child run's detail page won't show the parent until after persistence.

5. **driven smokes not run (owner-pending per P5 gate protocol).** The P5 plan requires `run-shamshir` skill driven smokes for each UI fix. These were deferred as OWNER-PENDING per the protocol — structural verification + unit test pass gates the commit. ## Risks for next phase (P6 — wild list)

1. **The idempotency lock is a `lock(this._idempotencyLock)` inside `StartAsync`.** `Start()` (private) is called inside the lock. `Start()` accesses `_runs` (ConcurrentDictionary, thread-safe) and spawns a `Task.Run`. The lock is held only during the synchronous `Start` call + dictionary mutation — sub-millisecond. Risk of lock contention is negligible, but if `Start` ever becomes async or long-running, the lock becomes dangerous.

2. **`completed-with-warnings` status chip relies on comparing the exact string `"completed-with-warnings"` at multiple layers** (C# `RunStatusResolver`, `BuildProgress`, `GetCurrentProgress`, Angular `statusVariant`, `statusLabel`). If the status constant ever changes, 4+ locations need updating. No centralized constant or enum for terminal status strings. Consider extracting to `RunStatus` constants in future.

3. **The `groupedRuns` computed in the run list O(n²) in worst case** — for every run, it filters sibling arrays. With 50 runs (the list limit), this is trivial. With 5000 runs, it would be ~25M operations. The take(50) limit in `GetRunsAsync` bounds this, but if the limit is ever raised, reconsider the grouping algorithm.

4. **P6 compare-both reconciliation depends on `ComparePairId` being consistently set on both legs.** Verify that `RunCompareBothAsync` correctly stamps the shared ID on both tape parent and cTrader child before persistence. The F16 code only adds read-side plumbing — the write side was already in place from P6.1.
