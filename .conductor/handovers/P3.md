## Bugs found and fixed (this audit)

### 1. ApplyCalibrationAsync stale-pass (FIXED — `HttpStepRunner.cs`)
- **Root cause:** The `ApplyCalibrationAsync` method called the API but discarded the response JSON, always returning `StepOutcome.Pass(applied: true)`. If the calibration endpoint returned `{"saved":false}` or an error, the step would incorrectly report success.
- **Fix:** Parse the response for `saved: true`; FAIL if absent or false. Added `ParseBool` / `ParseString` helper methods.
- **Test:** Covered by existing executor failure tests (step fails → pipeline fails).

### 2. Report step was a no-op (FIXED — `HttpStepRunner.cs`)
- **Root cause:** The `Report` step only emitted context values as verdict fields — no artifact, no file, no narrative. The plan specifies "research report --pipeline <id> writes markdown artifact."
- **Fix:** Report now writes `report.md` (pipeline ID, timestamp, context-value table) into the artifact dir and returns the file path in `StepOutcome.ArtifactPath`. Made async to support file I/O.
- **Test:** Covered by executor completion test (report step passes, pipeline completes).

### 3. Artifact dir silently null (FIXED — `PlaybookExecutor.cs`)
- **Root cause:** When `--artifact-dir` was not passed, `artifactDir` was null, and `WriteArtifactAsync` silently returned null — all reconcile JSONs, reports, and diagnostics were discarded.
- **Fix:** Executor now auto-creates `research-artifacts/{pipeline-id:N}/` under the current directory when no dir is specified. Added safety warning to `WriteArtifactAsync` if dir is still null.
- **Tests:** `Executor_AutoCreatesArtifactDir_WhenNoneGiven`, `Executor_UsesUserSuppliedArtifactDir_WhenGiven` (both added this session, green).

## Shortcuts, weaknesses, and not-fully-covered areas

### 1. cTrader child run ID for reconcile (NOT FIXED — blocked by credentials)
The `venue-parity.json` playbook uses `compareBoth: true` which creates paired tape + cTrader runs. The `start-run` step records only one runId. The `reconcile` step references `right: "ctraderRun"` but that context key is never set. The `owner-gate` step's comment says "supply the ctrader run id" but there is no mechanism:
- The server's compare-both response may include the child ID (unverified).
- The CLI's `StartRunPlan.ParseStartResponse` only extracts one runId.
- A follow-up GET on the parent run could discover the child ID.
- **Deferred** until a real compare-both run is possible (needs cTrader credentials).

### 2. HTTP endpoints not exercised end-to-end
The CLI verbs call these live API endpoints, but none have been verified against the running app:
- `api/exit-lab/evaluate` (ExitLabAsync)
- `api/exit-lab/calibrations` (ApplyCalibrationAsync — now parses response, untested live)
- `api/walk-forward/start` (WalkForwardAsync)
- `api/backtest/analytics/reconcile` (ReconcileAsync)
- Unit tests verify the *parsing* of responses (pure helpers), not the *delivery*.

### 3. No cancellation token propagation through artifact writes
`WriteArtifactAsync` in `Report` passes `CancellationToken.None`. If the pipeline is cancelled mid-report, the file write can't be interrupted. Low impact — report is always the last step.

### 4. No concurrency guard on pipeline status updates
The executor and the UI approve/reject share the same `Update`/`UpdateStep` endpoints without optimistic concurrency tokens. The CLI is single-threaded and the UI only acts on parked owner-gates, so races are unlikely. Still, two CLIs running the same `--resume` concurrently could produce interleaved step states.

### 5. ParseBool/ParseString are private to HttpStepRunner
These could be extracted to a shared pure helper for testability. Currently tested indirectly via executor + fake-runner pattern. Not a bug — existing code conventions keep helpers in their consuming class.

### 6. WalkForwardAsync / ExitLabAsync do not await completion
Both steps start a job but don't poll for completion (unlike `AwaitRunAsync`). The playbook's next step (`assert-gates` on the run, or `owner-gate`) provides the natural wait. Acceptable for the sequential playbook model.

### 7. ExitLabController.SaveCalibration uses DateTime.UtcNow
`ExitLabController.cs:179-181` uses `DateTime.UtcNow` for `FittedAtUtc`, `CreatedAtUtc`, `UpdatedAtUtc`. This is in the web layer (not engine/decision time), consistent with peer controllers and the pre-existing convention — not a violation.

## Risks for the next phase (P4 / P4.1)

1. **F11 data famine:** The `explore-exit` playbook is designed to populate `TradeExcursions` but has never been run end-to-end. Without a complete exploration run producing real excursion rows, the exit-lab grid will show zero cells and the pipeline will report `VERDICT: FAIL cells=0 error=no-cells`. Entry to P4 depends on the exploration funnel having data.

2. **cTrader credentials:** Every owner-gate (P2.2, P3.4 LIVE) requires cTrader credentials to resolve. The pipeline engine and UI are credential-free proven — the blocking item is not code quality but operational access.

3. **Config drift (F9):** The venue-parity playbook uses `compareBoth: true` which exercises the compare-both code path. Bugs B1–B3 (from the AGENTS.md struggle log) were fixed in the orchestrator but never end-to-end verified. The first live compare-both run may uncover new issues.

4. **`exitlab-eval` step params are sent verbatim as the body** — if the playbook author includes extra keys (like `as` for context reference), the server would receive unexpected fields. Currently the explore-exit playbook has clean params, but this is fragile.
