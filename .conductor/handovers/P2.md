## 2. Bugs found by the audit — all FIXED this session (with tests where testable)

1. **False `ILLEGAL transition` audit-trail noise on cancel-during-finalize.** The happy path enters `finalizing`;
   if an `OperationCanceled` then lands (cancel/timeout while finalizing), `RunAsync`'s OCE handler re-entered
   `finalizing` via `TransitionRun(state, Finalizing)`. A self-transition is not a legal edge, so the old
   `TransitionRun` logged `LogWarning` **and wrote a `LIFECYCLE` journal row** claiming an illegal jump —
   polluting the exact audit trail that flag is meant to reserve for genuine ordering bugs.
   **Fix (`b7b15cb`):** added pure `RunStateMachine.Classify(from,to) → {Legal, IdempotentNoOp, Illegal}`.
   Self-transition (re-enter current state) and leaving a terminal (double-cancel / post-completion teardown) are
   `IdempotentNoOp` (debug only, no journal); only a real violation from a live non-terminal state is `Illegal`.
   `CanTransition`/`TryTransition` unchanged (ratchet — the `SelfTransition_IsNotLegal` truth test stays green).
   **Tests:** +20 `Classify` cases incl. `finalizing→finalizing → IdempotentNoOp` and a full state×state
   `Classify_NeverContradicts_CanTransition` matrix.

2. **`Cancel()` blocked the web-request thread.** `KillCtraderProcessTreeAsync` was declared `async` but had **no
   `await` before** its synchronous `Process.GetProcessesByName` + `proc.Kill(entireProcessTree)` loop (the only
   await was `await Task.CompletedTask` at the very end). The cancel path's fire-and-forget
   `_ = KillCtraderProcessTreeAsync(...)` therefore ran the whole kill loop **inline on the caller** (the Cancel
   API thread). **Fix (`b7b15cb`):** the method now offloads via `Task.Run`; the post-run reaper still awaits it.
   *No automated test* (killing real OS process trees is not unit-testable here) — change is by inspection.

3. **`research run await` leaked a non-deterministic timeout verdict.** If the overall budget elapsed while a poll
   `GET` was in flight, the `TaskCanceledException` propagated past the loop to the top-level handler, emitting
   `VERDICT: FAIL error=TaskCanceledException` (exit 3) instead of the contract's `error=await-timeout` (exit 1).
   An agent branches on the verdict key. **Fix (`bce458d`):** catch OCE (when cts cancelled) around the poll GET
   and fall through to the single deterministic `await-timeout` verdict. Covered by reasoning, not a live test
   (the await loop is top-level `Program.cs`; the pure units it depends on are already tested).

## 3. What is SHORTCUT / WEAK / ASSUMED / NOT COVERED (do not oversell)

- **P2.2 IS NOT IMPLEMENTED.** Auto-promoted to DONE (OWNER-PENDING). Concretely missing:
  - **cTrader run QUEUE (Q2): does not exist.** `RunStateMachine.Queued` is defined but **nothing enqueues** and
    there is no "at most one cTrader run" serialization. `Start()` fires `RunAsync` immediately regardless of venue.
    A second concurrent cTrader run would still race on ports/DB. Source-verified (no queue type in the orchestrator).
  - **compare-both child row up-front (F16): not done — and there is a real persistence gap.** In
    `RunCompareBothAsync` the cTrader child is registered **in-memory only** (`_runs[ctraderRunId] = …`, ~line 953)
    with **no `WriteStartRecordAsync`**. A DB row is written **only on the success branch** (`WriteEndRecordAsync`,
    ~line 979). So a **cancelled or failed compare-both cTrader leg leaves NO persisted child row at all** — the
    precise F16 symptom ("only tape ran" is indistinguishable from "cTrader child failed silently"). Auto-reconcile
    on parent completion and `ReconcileJson`-on-parent are **not** implemented. (This is P2.2 scope, hence
    owner-pending — but flagged here so it is not lost.)
  - **The headline gate is UNRUN:** one real compare-both (EURUSD H1, 1 month) with a committed reconcile verdict
    requires cTrader credentials → OWNER-PENDING. Template lives in `docs/audit/RECONCILE-FINDINGS.md §P2.2`.
- **Watchdog (a P2.1 requirement) is STRUCTURAL ONLY, untested.** "CLI exit without completion ⇒ finalize within
  30s with real status" relies on the pre-existing 30s `BarStream.Completion` safety timeout + the P4.1 finally-net
  terminal write. **No new watchdog code and no fake-CLI-dies test** were added this phase. Live behaviours (real
  `ctrader-cli` tree kill on cancel; watchdog on CLI death) are unexercised — credential-gated.
- **The state machine is tested at the PURE layer only.** `TransitionRun` (the orchestrator integration) has **no
  direct automated test** — it is private and the orchestrator has no test harness. The cancel-during-finalize path
  that produced bug #1 is reasoned-through, not reproduced by a test.
- **ResearchCli is P3.1 FOUNDATION only.** Verbs: `run validate`, `run await`, `reconcile` (GET-only client).
  Not implemented: the **diagnostics bundle** design rule (on failure it emits a one-line `DIAG: <ExceptionType>`,
  *not* runId/status/last-50-journal/warnings); `reconcile` does **not** evaluate tolerances (it PASSes whenever the
  endpoint returns 200); `--forbid-warning-code` accepts only **one** code via CLI (dict overwrite); warning-code
  matching is a raw-substring `Contains` (brittle); `CliArgs` has a `--flag value` ambiguity (a flag immediately
  followed by a non-`--` token is parsed as an option — safe for the documented usages, latent otherwise).

## 5. Risks the NEXT phase (P3) should watch

1. **Sequence risk:** P3's canonical `venue-parity.json` playbook drives compare-both → reconcile, but compare-both
   child rows + auto-reconcile (P2.2) do **not** exist yet. The playbook engine can be built, but that playbook
   cannot be exercised end-to-end until P2.2 lands. Don't claim it green off a tape-only run.
2. **ResearchCli client only GETs.** P3 needs POST verbs (`data ensure`, `run start [--compare-both]`). The
   `api/runs/{id}` + reconcile shapes are assumed camelCase and un-versioned; verify against the live app.
3. **FLAKY integration test (pre-existing, not P2):** a `WebApplicationFactory` temp-DB teardown race
   ("SQLite Error 14: unable to open database file" while a background run finalizes after the test deleted its
   temp DB). Observed **1 failure once** this session; passed clean on immediate re-run **and** on the pre-change
   baseline. Not caused by P2 changes, but it can turn the Integration gate red intermittently — **re-run before
   treating as a regression.** A real fix would await the run task / `MarkCompleted` before disposing the factory DB.
4. **OCE-path honesty gap (out of P2.1 scope):** a run truncated by the 30-min linked timeout (not a user cancel)
   is reported `completed` (`userCancelled=false → Completed`). Arguably should be `completed-with-warnings`.
   Unchanged this phase (needs a distinct timeout signal); fold into P5 status-truth work.
