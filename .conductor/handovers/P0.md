## 3. What this audit CHANGED

Commit `305a853` — ratchet-only (adds warnings/observability; no gate/golden/truth-file weakening):

1. **F6-R detection safety net (option b).** The P0.3 QA correctly flagged that the barrier only
   backfills from `PublishTradeClosed` effects, so the *audited* crashed-teardown run `f7b0538d` (whose
   7 closes exist ONLY as `OrderFilled` close-fill events, not as effects) computed `expected=0`, raised
   no warning, and still reported a silent `TotalTrades=0` — the exact lie F6 exists to prevent.
   The barrier now also counts venue close-fills (`OrderFilled` EventJson with a non-null `CloseReason`)
   and exposes `Unreconstructable = !Failed && Persisted+Backfilled==0 && JournalCloseFills>0`; the
   orchestrator attaches `TRADES_UNRECONSTRUCTABLE:{n}` → `completed-with-warnings`.
   - **Evidence-backed + false-positive-free.** Verified by direct SQL against the kept audit DB across
     all 6 paired runs (table in `evidence/P0.3-…md §7`): only `f7b0538d` (close-fills 7 / effects 0 /
     trades 0) trips it. Healthy runs always have effects>0 or trades>0. `817af3f5`'s extra 2 close-fills
     (26 vs 24) do NOT trip it because its trades persisted.
   - Tests: `CrashedTeardown_CloseFillsButNoPublishTradeClosed_IsUnreconstructable` and
     `HealthyRun_ExtraCloseFill_ButTradesPersisted_IsNotUnreconstructable`.
2. **F1 drift-log accuracy.** The drift warning logged `_runContext.InitialBalance`, which is 0 when the
   balance arrived via the RiskManager (older callers) — so it could print `ConfigBalance=0` while drift
   was real. Now logs the resolved balance. Cosmetic/observability only; no behavior change.

## 4. What is SHORTCUT / WEAK / ASSUMED / not fully covered (do not oversell)

- **F6-R economics NOT recovered (deferred — HUMAN decision).** The safety net DETECTS an unreconstructable
  loss and reports it honestly, but it does NOT rebuild the trade's economics. Doing so means pairing the
  open-fill and close-fill `OrderFilled` events by `OrderId` and synthesizing a `TradeResult` — that
  touches the cTrader VenueManaged reconcile-close mapping (kernel/adapter-adjacent, a STOP condition).
  Recommended follow-up (option a): have the VenueManaged close path emit `PublishTradeClosed` into the
  journal before teardown, so the existing backfill recovers economics for free. Needs an owner call.
- **All P0 real cTrader/paired runs are OWNER-PENDING (creds-gated).** Every P0 checkpoint is proven
  credential-free (FakeTransport, pure functions, the kept audit DB). NONE has been confirmed on a live
  paired run against the post-P0 build. Specifically unconfirmed on real runs: (1) equal lots in a live
  paired DB (F1); (2) 3 consecutive headless cTrader runs ending `completed` with zero NetMQPoller
  messages (F5); (3) a real BTC-scenario producing `TRADES_LOST`/`TRADES_UNRECONSTRUCTABLE` +
  `completed-with-warnings` (F6/F6-R). The P2.2 owner-gate is where these get exercised end-to-end. Do
  NOT attempt live runs to "verify" — auto-promote and continue per orchestrator policy.
- **F2 is measurement only (by design, Q4).** The +1-decision-bar cTrader lag is quantified, not fixed.
  Any actual latency fix (M1-cadence command drain in the cBot) is deliberately out of P0.
- **Status-string duplication.** `RunQueryService.GetRunsAsync` derives status inline (EF-translatable
  LINQ can't call `RunStatusResolver.Resolve`) while every other reader uses the resolver. Behaviour
  matches today, but it is a drift risk (e.g. the inline check does not treat the literal string `"null"`
  as empty, the resolver does). Low priority; fold into P5's run-store consolidation.
- **Root-container cross-rate for backfill uses seeded defaults**, not the per-run live `CrossRateStore`.
  Fine for currency-tagging (venue trades carry their own gross/net), but a backfilled trade on an exotic
  cross could mis-tag currency in the rare no-venue-PnL path. Not hit by any current test data.
