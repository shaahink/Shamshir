## 2. What is shortcut / weak / assumed / not fully covered (honest)

1. **Baseline branch masks a disk≠DB divergence at the M42 upgrade boundary.** In `ConfigSyncService.Classify`,
   when `SeededHash` is null (first run after M42), the file is *baselined* (hash recorded, content NOT
   applied) regardless of whether the file already differs from the DB row. This is the deliberate
   conservative choice — with no `SeededAtUtc` yet we cannot tell a genuine pending edit from a UI hand-edit,
   and applying would risk clobbering a hand-edit. Cost: a real pending file edit present at the exact moment
   of the M42 upgrade needs ONE more file-touch to propagate. **Neutralized in the shipped state** because Q1
   reverted the 8 JSONs to Market so disk==DB at the boundary. Deferred by design, not a regression. No test
   pins this behavior yet — a follow-up could add one documenting it.

2. **`DateTime.UtcNow` in new code vs the IEngineClock meta-rule.** `ConfigSyncService` (4 sites: SeededAtUtc
   stamping) and `ReferenceScalePopulator.RefreshedAtUtc` use `DateTime.UtcNow` directly. Accepted deviation,
   NOT changed: (a) no `IEngineClock` is registered in the Web root scope — `AuditStampInterceptor` itself
   falls back to `DateTime.UtcNow` there (`ServiceRegistration.cs:69`), as do the existing `SqliteStrategyConfigStore`/
   `SqliteRiskProfileStore` upserts; (b) these are startup/maintenance infra paths, not the deterministic
   engine decision path the rule protects. Injecting a clock now would be no-op churn AND could *desync*
   `UpdatedAtUtc` (interceptor) from `SeededAtUtc` (direct) if a simulated clock is ever registered — which
   would break the 5-second hand-edit tolerance. Flagged for visibility.

3. **`ReferenceScaleCli` builds its own ServiceProvider without `SqlitePragmaInterceptor`** (no `busy_timeout`).
   If `compute-reference-scales` is run WHILE the Web app is live it could hit `SQLITE_BUSY` on write. It is a
   standalone maintenance verb (owner stops the app first), so low risk — but not hardened for concurrency.

4. **`DbPathResolver.FindRepoRoot` fallback is dev-only.** It walks up for `.slnx`/`.git`; in a published/
   deployed layout with neither marker it silently falls back to a fixed five-levels-up path. Fine for the
   current dev workflow; would misfire if this is ever shipped as a self-contained deployment.
