## 2. Root cause + fix

`EngineRunner.RunAsync` startup reconciliation (was `EngineRunner.cs:114-124`) adopted the venue's
hello-reported balance into `initialBalance` whenever `> 0`, in ALL modes including backtest. The
cTrader cBot hello reports the demo account's real balance (~25k), so the cTrader leg's kernel sized
off 25k while tape (replay/`GetAccountStateAsync` returns the configured balance) sized off 100k.

Fix: `EngineRunner.ResolveInitialBalance(mode, configBalance, riskManagerBalance, venueBalance)` — a
pure, unit-tested function. In **backtest** the configured balance is authoritative; a disagreeing
venue balance is logged as drift (`Startup balance drift: … NOT adopted (F1 parity guard)`) but never
adopted. **Live/paper** keep venue adoption. Config balance is plumbed
`cfg.Balance → EngineHostOptions.InitialBalance → EngineRunContext.InitialBalance → EngineRunner`.

`VenueSizingParityTests.Fixed_BothVenues_ResolveToConfigBalance_AndSizeEqually` (R3 after-the-fix):
with resolution corrected, both venues resolve to 100k ⇒ **equal Lots (1.92) and equal RiskAmount**.
`Live_AdoptsVenueBalance` proves live mode is unchanged (adopts 25k, no drift).

## 3. Mechanism SOURCE (no creds)

`CtraderHello_SurfacesDemoBalance_ThatBacktestMustNotAdopt`: a `FakeMessageTransport` cTrader hello

exact stale value `EngineRunner` used to adopt. `ResolveInitialBalance(Backtest, 100k, 0, 25k)` →
InitialBalance=100k, HasDrift=true.
