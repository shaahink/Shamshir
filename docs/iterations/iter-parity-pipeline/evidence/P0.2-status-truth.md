# P0.2 — Evidence: run-status truth + the disposed-NetMQPoller teardown crash (F5)

Two commits (one subphase each, R4):
- **P0.2b** `6533c7e` — root-cause fix: idempotent `NetMqMessageTransport.DisconnectAsync`.
- **P0.2a** `<this commit>` — design fix (Q5): separate engine-result from transport-teardown so a
  complete run with a teardown fault is `completed-with-warnings`, never `failed`.

## 1. F5 root cause (R3 — one observed repro before, one clean after)

AUDIT F5: all 4 cTrader runs saved `ExitCode=1`,
`ErrorMessage='Cannot access a disposed object. Object name: NetMQPoller'` despite complete stats. B4
(unsubscribe handlers) did NOT prevent it.

**Standalone repro (credential-free):** a bare `NetMQPoller` — `Stop(); Dispose();` then `Stop()` again
throws `ObjectDisposedException: 'NetMQPoller'` — the exact F5 message.

```
first disconnect OK
SECOND Stop THREW: ObjectDisposedException: Cannot access a disposed object.
Object name: 'NetMQPoller'.
```

**Why it fired every run:** `NetMqMessageTransport.DisconnectAsync` was non-idempotent and is invoked
TWICE per cTrader run — the `BarStream` safety-net force-disconnect (`BacktestOrchestrator.cs:1375`)
then host disposal (`:1382` → `CTraderBrokerAdapter.DisposeAsync` → `transport.DisconnectAsync`). The
2nd call hit `_poller.Stop()` on the disposed poller and threw; the exception propagated out of
`RunEngineNetMqAsync`'s `finally` into `RunAsync`'s outer catch (`:468`), which stamped a COMPLETE run
`failed` (status = `ErrorMessage != null ? failed : completed`).

**In-suite repro (R3), `NetMqTransportTeardownTests` — real loopback sockets, NO cTrader creds:**
- BEFORE fix (transport stash-reverted): 2/2 FAIL with
  `ObjectDisposedException: NetMQPoller at NetMQPoller.Stop() (NetMqMessageTransport.cs:89)`.
- AFTER fix: 2/2 PASS.

Fix: single-shot `Interlocked.Exchange(ref _teardownStarted, 1)` guard — the second `DisconnectAsync`
is a no-op.

## 2. Status truth (Q5) — the design fix under the bug

Even with the race fixed, a teardown fault must never overwrite a complete result. New vocabulary:
`completed-with-warnings` (Q5). `failed` is reserved for "no trustworthy result".

- New `WarningsJson` column on `BacktestRuns` (EF migration `M41_RunWarnings`, additive TEXT, nullable).
- `RunStatusResolver` (Domain, pure) is the single source of truth, replacing 4 divergent inline
  `ErrorMessage != null ? failed : completed` derivations (RunQueryService list+detail,
  BacktestQueryService, BacktestAnalyticsController).
- `RunEngineNetMqAsync`'s teardown `finally` now isolates each step (`SafeTeardownStepAsync`) and records
  a `RunWarning` instead of letting an exception propagate — defense-in-depth beyond the transport fix.
- Finalization: a successful result with warnings ⇒ `completed-with-warnings`; `ErrorMessage` stays null.
