# P3.2a — Research pipeline persistence (Q6 / F8-adjacent lifecycle)

**Checkpoint:** P3.2 (persistence half) — `ResearchPipelines` + `ResearchPipelineSteps` DB tables +
CRUD/review API. Pipeline state lives in the running app's single DB (Q6, F10), resumable by id,
reviewable by the UI (P3.3).

## What landed

- Entities `ResearchPipelineEntity` + `ResearchPipelineStepEntity`
  (`src/TradingEngine.Infrastructure/Persistence/Entities/ResearchPipelineEntities.cs`), both
  `IAuditableEntity` (D5) — audit arch test lists only the pre-existing `ExitCalibrationEntity`, not these.
- Mapping `ResearchPipelineMappings.cs`: steps cascade off the pipeline; `(PipelineId, StepIndex)` unique.
- Migration `20260708173645_M43_ResearchPipelines` (+ Designer + snapshot).
- API `ResearchPipelinesController` (`/api/research/pipelines`): GET list, GET {id}, POST create,
  PUT {id}, PUT {id}/steps/{index}, POST {id}/approve, POST {id}/reject. The CLI executor (P3.2b)
  persists all state through these — no direct DB access from the CLI (Q3).

## Notes / traps

- `EntityAuditableTests` still red on `ExitCalibrationEntity` ONLY — pre-existing (tracker trap #1),
  NOT introduced here; the two new entities are compliant.
- The controller uses `DateTime.UtcNow` for row create/complete timestamps — matches the existing
  controller convention (WalkForward/Experiments); the AuditStampInterceptor also stamps audit cols.
  `IEngineClock` is the rule for ENGINE/decision time; these are wall-clock request timestamps in the
  web layer, consistent with every peer controller.
- Live API round-trip (start app, POST/GET pipelines) is owner/next-session (needs app running).
