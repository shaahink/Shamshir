# P1.1 — One database (F10) — evidence

**Checkpoint:** P1.1 (iter-parity-pipeline PLAN §4). **Finding:** AUDIT F10 (two databases; Host CLI dead
on arrival against the un-migrated root `data/trading.db`) + R9 (environment drift unnoticed).
**Date:** 2026-07-08. **Branch:** `iter/parity-pipeline`.

## What changed

1. **`DbPathResolver`** (`src/TradingEngine.Infrastructure/Persistence/DbPathResolver.cs`) — single source of
   truth for the trading/market-data DB paths, anchored to the **repo root** (marker `TradingEngine.slnx`
   or `.git`), NOT the process cwd. Canonical trading DB = `<root>/src/TradingEngine.Web/data/trading.db`.
   Rooted config values pass through; relative values resolve against the repo root.
2. **Wired** into the Web app (`ServiceRegistration.AddPersistence`), the backtest orchestrator
   (`BacktestOrchestrator.RunEngineNetMqAsync`), and the Host CLI (`Program.cs`). Removed the relative
   `Persistence:DbPath` key from `appsettings.Development.json` (the canonical default now resolves to the
   same physical file the 18 MB audit DB already lives at — no move, no data loss).
3. **`MigrationGuard`** (`src/TradingEngine.Infrastructure/Persistence/MigrationGuard.cs`) — the Host CLI
   fails loud (`PendingMigrationsException`, exit ≠ 0) with the exact DB path when the opened DB has
   pending migrations. It never migrates (the Web app owns migration application).
4. **`compute-reference-scales`** Host CLI verb (`ReferenceScaleCli.cs`) — builds a minimal provider
   (the full engine host does not register the market-data store), runs the migration guard, then
   populates the 84 `ReferenceScales` cells into the SAME DB the Web app uses. Closes the
   iter-quant-model carry-forward that was blocked by F10.
