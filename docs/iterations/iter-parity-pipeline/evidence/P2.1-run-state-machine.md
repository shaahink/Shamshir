# P2.1 (F8) — Run lifecycle state machine — evidence

**Commit:** `ccf6aa4` (feat P2.1) · **Session:** s10 · **Date:** 2026-07-08

## What F8 was

The audited F8 finding: the orchestrator's run status was a stringly-typed, **multi-writer** property
(`state.Status = "running"` scattered across the happy / cancel / OperationCanceled / exception paths)
with **no enforcement** of legal ordering and **no tests**. Cancel was brittle; "stuck running forever"
was a whole bug class.

## What landed

1. **`src/TradingEngine.Domain/RunStateMachine.cs`** — pure, no infra deps. Enumerates
   `queued → starting → running → finalizing → completed | completed-with-warnings | cancelled | failed`
   with an explicit allowed-transition map. `TryTransition` never throws; leaving a terminal state is a
   benign no-op (double-cancel), an illegal jump from a non-terminal state is rejected + status unchanged.
   Shares the four persisted terminal strings with `RunStatusResolver` (one vocabulary, Q5).

2. **`BacktestOrchestrator.TransitionRun(state, to)`** — the SINGLE guarded writer. Every status change
   in `RunAsync` + `RunCompareBothAsync` routes through it (`grep '.Status = '` → 1 hit, inside
   `TransitionRun`). The happy path now goes **running → finalizing → terminal** (a direct
   running→completed jump is forbidden, so the trade-persistence barrier/stats/end-record step can never
   be skipped). `finalizing` is transient/in-memory only — `WriteEndRecord` runs *after* the terminal
   transition, so a persisted row is always one of the four terminals.

3. **Cancel hardening (F8):** `Cancel` is idempotent + truthful — records `CancelRequested` intent + trips
   the token instead of stamping a lying `cancelled` mid-finalize; a double-cancel / cancel-of-terminal is
   a no-op. On a live cTrader run it best-effort kills the ctrader-cli process tree (extracted
   `KillCtraderProcessTreeAsync`, reused by the post-run reaper). Each reap is journalled to VenueSessions.

4. **Observability (R7):** illegal transitions → `LogWarning` + journal `LIFECYCLE`; every transition
   logs `From->To` at debug; every reap logs pid+reason.

## OWNER-PENDING (needs cTrader credentials — folded into P2.2)

The state-machine **transitions** for cancel/double-cancel/cancel-during-finalize are unit-tested
credential-free. The **live** behaviors — process-tree kill of a real `ctrader-cli`, and the watchdog
finalizing a run within 30s on CLI-exit-without-completion — require a real cTrader run to exercise
end-to-end. The 30s bar-stream safety timeout + P4.1 finally-net terminal write already implement the
watchdog structurally (`RunEngineNetMqAsync` finally block); a live fake-CLI-dies test is the P2.2 gate.
