# P0.3 — Evidence: trade-persistence integrity barrier (F6)

One subphase = one commit (R4). Artifacts produced by tests run THIS phase (no cTrader creds needed):
`dotnet test tests/TradingEngine.Tests.Integration --filter "FullyQualifiedName~TradePersistenceBarrier"` → 3/3.

## 1. F6 mechanism + the barrier (R3 — observed before + observed after)

AUDIT F6: BTC cTrader run `f7b0538d` journalled 12 OrderProposed + 17 OrderFilled + **7 closes** yet
persisted **0 TradeResults** and reported `TotalTrades=0` — the close→`PublishTradeClosed`→TradeResult
path lost everything before finalization (VenueManaged closes not mapped back and/or the
`TradePersistenceHandler` channel never flushed before the run ended). Finalization had **no integrity
barrier** ("journal says N closed positions, TradeResults says M — refuse to report M silently").

**The barrier** (`TradePersistenceBarrier.ReconcileAndBackfillAsync`, Infrastructure):
1. Streams the run's journal, collects every `PublishTradeClosed` effect from `EffectsJson` (parallel to
   `EffectKinds`) — the lossless source of truth. This is `expected`.
2. Counts persisted `TradeResults` rows for the run — this is `persisted`.
3. On a shortfall, backfills each missing close by reconstructing the `TradeResult` via
   `TradeResultFactory.FromClose` — the SAME builder the live path uses — and returns `(Expected,
   Persisted, Backfilled)`.

**R3 repro (in-suite, real SQLite):** `BtcScenario_JournalHasCloses_TradeResultsEmpty_BackfillsAll`
- BEFORE: journal has 3 closes, `db.Trades` count = **0** (the exact F6 silent loss).
- AFTER: `Expected=3, Persisted=0, Backfilled=3`; `db.Trades` count = **3**, reconstructed faithfully
  (NetPnL 770 = gross 800 − 30 commission; SL close −530). Not `TotalTrades=0`.

Two more barrier tests: `PartialPersistence_BackfillsOnlyTheMissing_NoDuplicates` (multiset key on
PositionId|ClosedTicks|Reason|Lots → backfills only the missing SL, no dup of the persisted TP) and
`FullyPersisted_NoBackfill_NoLoss` (idempotent: HasLoss=false, no warning on a clean run).

## 2. Anti-drift: one construction path (TradeResultFactory)

`EffectExecutor.HandlePublishTradeClosed` construction logic was extracted **verbatim** to
`TradeResultFactory.FromClose` (Services). Live path and backfill now build trades identically — a
recovered trade is byte-for-byte what the live close would have written. Only caller-supplied field is
`timeframe` (live resolves from strategy registry; backfill = null, the journal carries no TF).
**Golden held byte-identical** across the extraction (see §4), proving the refactor is faithful.

## 3. Wiring → completed-with-warnings (reuses P0.2 plumbing, R7)

`BacktestOrchestrator.RunAsync`: after a successful run, BEFORE `GetTradeStatsAsync`,
`RunTradePersistenceBarrierAsync` runs the barrier (backfill first so stats count restored trades). A
shortfall calls `AddTeardownWarning(runId, "TRADES_LOST:{expected}:{persisted}", …)` — the exact P0.2
WarningsJson machinery → the run finalizes **`completed-with-warnings`**, never a silent lower
TotalTrades and never `failed`. The barrier logs `TRADE_BACKFILLED` / `TRADES_LOST` (Serilog templates,
R7). Barrier failures are themselves caught → `TRADE_BARRIER_FAILED` warning; never breaks finalization.

## 7. P0 AUDIT hardening — F6-R detection safety net (option b), 2026-07-08

The P0.3 QA (s5) found the barrier fixed the persistence-channel-loss case but NOT the audited
crashed-teardown case: `f7b0538d` journalled its 7 closes as venue **OrderFilled close-fill events**, not
as `PublishTradeClosed` **effects**, so the barrier computed `expected=0` → no `TRADES_LOST` warning →
still a silent `TotalTrades=0`. Evidence (direct SQL against the kept audit DB
`src/TradingEngine.Web/data/trading.db`, run this audit):

| Run | Venue | close-fills (OrderFilled w/ CloseReason) | PublishTradeClosed effects | TradeResults |
|---|---|---|---|---|
| `f7b0538d` | ctrader | **7** | **0** | **0**  ← the F6-R silent loss |
| `817af3f5` | ctrader | 26 | 24 | 24 |
| `44175d3e` | ctrader | 3 | 3 | 3 |
| `81729685` | ctrader | 7 | 7 | 7 |
| `2c9551d1` | tape | 28 | 28 | 28 |
| `020fd4eb` | tape | 8 | 8 | 8 |

**The safety net (detection-only, tracker option b):** the barrier now also counts venue close-fills
(`OrderFilled` events whose EventJson carries a non-null `CloseReason`) and exposes
`Unreconstructable = !Failed && Persisted + Backfilled == 0 && JournalCloseFills > 0`. When true the
orchestrator attaches a `TRADES_UNRECONSTRUCTABLE:{n}` warning → `completed-with-warnings`. This makes
the `f7b0538d` signature honest instead of a silent zero. **False-positive-free** (proven against all 6
runs above): every healthy run has `PublishTradeClosed > 0` or persisted trades, so `Persisted+Backfilled`
is never 0 for them — only `f7b0538d` (0/0/7) trips it. `817af3f5`'s extra 2 close-fills (26 vs 24) do
NOT trip it because its trades persisted.

**What this does NOT do (deferred, owner-decision — option a):** it does not RECONSTRUCT the economics
of an unreconstructable trade. Rebuilding a `TradeResult` from an open-fill + close-fill pair touches the
cTrader VenueManaged reconcile-close mapping (a kernel/adapter-adjacent STOP condition) and needs an
owner call. See the HUMAN line in TRACKER handoff.

**Tests (Integration):** `CrashedTeardown_CloseFillsButNoPublishTradeClosed_IsUnreconstructable`
(3 close-fills, 0 effects, 0 trades → `Unreconstructable=true`, `HasLoss=false`, still 0 TradeResults)
and `HealthyRun_ExtraCloseFill_ButTradesPersisted_IsNotUnreconstructable` (the 817af3f5 false-positive
guard). Barrier suite now **5/5**; golden unchanged (Domain/Infrastructure-only, no kernel decision path).
