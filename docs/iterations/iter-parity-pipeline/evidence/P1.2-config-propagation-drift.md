# P1.2 ‚Äî Config propagation + drift (F9, F7) ‚Äî evidence

**Checkpoint:** P1.2 (iter-parity-pipeline PLAN ¬ß4). **Findings:** AUDIT F9 (the strategy-config DB ignored
the agent's JSON edits ‚Äî the limit-order rollout was inert), F7 (per-run effective config), R5 (config
two-sources-of-truth blindness). **Date:** 2026-07-08. **Branch:** `iter/parity-pipeline`.

## What changed

1. **Schema M42_ConfigSeedHash** ‚Äî adds nullable `SeededHash` + `SeededAtUtc` to `StrategyConfigs` and
   `RiskProfiles`. `SeededHash` = SHA-256 of the source JSON at last sync-from-disk; `SeededAtUtc` = when.
2. **`ConfigSyncService`** (`src/TradingEngine.Infrastructure/Configuration/ConfigSyncService.cs`) ‚Äî runs on
   Web startup AFTER the one-time seeders. Per `config/strategies/*.json` + `config/risk-profiles/*.json`:
   hash the file, compare to `SeededHash`:
   - equal ‚Üí in-sync;
   - `SeededHash` null (first run after M42) ‚Üí baseline the hash, content untouched, no Version bump;
   - changed AND row NOT hand-edited (`UpdatedAtUtc ‚â§ SeededAtUtc + 5s`) ‚Üí re-apply JSON, bump Version, log;
   - changed AND hand-edited via UI ‚Üí **drift conflict: NOT clobbered**, surfaced by the endpoint.
   (5s tolerance because the audit interceptor re-stamps `UpdatedAtUtc` on the same save that writes
   `SeededAtUtc`; a real UI edit lands minutes/hours later.)
3. **`GET /api/system/config-drift`** (`SystemController`) ‚Äî read-only drift report (counts + per-item
   status + conflict list) for the Strategies-page chip (chip itself deferred to P5-UI).
4. **F7** ‚Äî `BacktestOrchestrator` now persists the effective config into `StrategyParamsJson` (was the dead
   literal `"{}"`) on BOTH the tape and cTrader write records. (`EffectiveConfigJson` already carried the
   truth for both venues ‚Äî verified in the audit DB: `orderEntry.method` present on ctrader runs ‚Äî so run
   config was already provable; this removes the misleading empty column the audit flagged.)

## Runtime evidence (R5) ‚Äî the real app against the unified DB

The seed-once trap (F9) is killed structurally: a JSON edit now propagates into the RUNTIME store on restart.

### 1. Real-app startup applies M42 + baselines (ASPNETCORE_URLS=:5177)

```
migration head -> 20260708152104_M42_ConfigSeedHash
StrategyConfigs (all 9): SeededHash set, SeededAtUtc set, Version=1 (baseline does NOT bump)
GET /api/system/config-drift -> 200 {"inSync":13,"conflicts":0,"hasDrift":false,...}   (9 strategies + 4 risk profiles)
```

### 2. Edit JSON ‚Üí restart ‚Üí DB reflects it (the F9 fix)

Edited `config/strategies/trend-breakout.json` orderEntry `"Market"` ‚Üí `"LimitOffset"`, restarted:

```
sqlite: SELECT Version, OrderEntryJson FROM StrategyConfigs WHERE Id='trend-breakout';
  2 | {"Method":1,"LimitOffsetPips":3,...,"LimitOffsetAtrFraction":0.15,...}
```

`Method` flipped 0‚Üí1 (Market‚ÜíLimitOffset) and `Version` bumped 1‚Üí2 ‚Äî **the JSON edit propagated into the DB**
(exactly what stayed inert before: every run the owner executed used Market because edits never propagated).

### 3. Revert JSON ‚Üí restart ‚Üí DB self-heals

Reverted the file to Market, restarted:

```
  3 | {"Method":0,...}          (back to Market, Version 1‚Üí2‚Üí3)
```

JSON is authoritative when the row was not hand-edited. Working tree left clean (`git status config/` empty);
DB integrity `ok`; ReferenceScales=84, BacktestRuns=9 intact.
