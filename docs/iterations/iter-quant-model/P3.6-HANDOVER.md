## 1. The spec (from PLAN.md:307)

> Record EVERY proposal (accepted, gated, and venue outcome: filled / expired-unfilled) with the signal
> price and time. For unfilled limit orders, the excursion recorder still tracks the counterfactual path
> from the signal (as if filled at signal close) for N bars. Output: per cell, fill-rate,
> expectancy-conditional-on-fill vs counterfactual-expectancy-of-missed — the honest answer to "should this
> strategy enter Market, Limit@0.25×ATR, Limit@0.5×ATR, or StopConfirm?". Entry tactic then becomes a sweep
> dimension in P4, per cell, instead of a hardcoded per-strategy choice.

## 3. Design questions / doubts (for owner review before implementation)

These are the unresolved design choices the implementing agent must resolve with the owner.

### Q1 — DecisionRecord.OrderId extension (small but intrusive)

**Trade-off:** Adding `Guid? OrderId = null` to `DecisionRecord` is 2 lines of code with zero call-site
breakage (default null). But it touches a domain type that's used in 25+ places across the kernel, engine
reducer, position lifecycle, and tests.

**Alternative:** Create a parallel event that carries `(OrderId, DecisionRecord)` and publish it
alongside `RecordDecisionEvent`. More code but zero domain-type changes.

**Recommendation:** Go with the `OrderId` extension — it's the cleanest correlation mechanism and the
`Guid?` default means existing tests compile and pass unchanged. The alternative creates an event that
duplicates `DecisionRecord`'s entire payload and would eventually need the same correlation anyway.

### Q2 — Where to hook venue expiry recording?

**Option A (recommended):** `KernelBacktestLoop.onExecution` callback (same pattern as `evaluateTrailing`,
`evaluateTimeFlatten`). Called inside `PumpAsync` before the execution event enters the kernel queue.
Pro: captures the raw venue outcome. Con: `KernelBacktestLoop` gains a new optional constructor parameter.

**Option B:** Hook in `EngineReducer.HandleOrderCancelled` by adding an `onCancelled` callback. Pro: stays
inside the engine. Con: requires threading a callback through `EngineReducer` which is currently a pure
reducer; breaks functional purity.

**Option C:** Post-hoc reconciliation after the run completes — correlate proposals with trades by
comparing OrderId lists. Pro: zero engine changes. Con: can't distinguish "limit expired" from "limit
was never tested" from "limit at correct price but already beyond bar range"; much lower fidelity;
counterfactual paths must be recorded speculatively for ALL limits.

**Recommendation:** Option A. The callback pattern already exists (`evaluateTrailing`, `_onBarProcessed`,
`_onEvent`) and is idiomatic in this codebase.

### Q3 — BacktestReplayAdapter: include counterfactuals or skip?

`BacktestReplayAdapter` has single-resolution bars (no M1 fine-bar loop). Counterfactual recording would
be one point per decision bar — coarse but honest.

**Recommendation:** Record counterfactuals in both adapters but clearly tag the fidelity level on the
entity (`Fidelity = "Tape"` vs `"Backtest"`). The UI differentiates tape-sourced cells (trusted) from
backtest-sourced cells (coarse, flagged). This keeps the code path unified.

### Q4 — EntryMethod grouping granularity

Should `LimitOffset` proposals be grouped by:
- (a) Raw method name only: `LimitOffset` (lumps 2-pip and 0.25×ATR together)
- (b) Normalized ATR fraction: `LimitOffset:0.25ATR` (default grouping)
- (c) Raw pips: `LimitOffset:2.0pip` (old configs before P2.6 units doctrine)

**Recommendation:** Group by `EntryMethod` + `EntryMethodDetail` where `Detail` is the resolved
ATR-fraction (from `UnitConversion`) when available, falling back to raw pips. The evaluator normalizes
into bins (e.g. `0.25×`, `0.5×`, `1.0×`) so slightly-different resolved values from different volatility
environments still group together.

### Q5 — Scope: all-at-once vs phased delivery

**Recommendation:** Deliver Phase A+B first (proposal ledger + recording hooks). Gate on:
- Unit tests for entity/repository round-trip
- Simulation test proving a tape run produces a populated `ProposalLedger` table with correct outcomes
- Manual smoke-test via `GET /api/runs/{id}/proposals`

Then decide on Phase C+E (counterfactual paths + UI) based on whether the per-cell fill-rate stats from
A+B alone provide enough signal. Phase C (counterfactual paths) is the most complex piece by a wide margin
(~300 lines in `TapeReplayAdapter` alone, plus kernel threading). If the fill-rate signal is weak without
counterfactual expectancy, proceed with C.
