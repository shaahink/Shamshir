## Root cause (final, evidence-backed)

Commit `9454878` (Jul 8 16:09, "P1.1 one database") **removed the `Persistence:DbPath` dev key**
from configuration. The Web app's own DbContext moved to `DbPathResolver` (canonical
`src/TradingEngine.Web/data/trading.db`), but `BacktestOrchestrator.RunEngineInProcessAsync`
kept its old fallback:

```csharp
var dbPath = _configuration.GetValue<string>("Persistence:DbPath")   // now null
    ?? Path.GetFullPath(Path.Combine(AppContext.BaseDirectory, "..","..","..","..","..", "data", "trading.db"));
```

From `bin/Debug/net8.0` that resolves to **repo-root `data/trading.db`** — a different file.
Result from Jul 8 16:09 → Jul 10 02:16: run rows + TotalBars (written by the Web's own context)
landed in the canonical DB, while the inner engine host wrote **journal / trades / equity to the
phantom root DB**. Hence the exact broken signature: `TotalBars=890, Journal=0, TradeResults=0,
EquitySnapshots=0, ExitCode=0`. cTrader runs were unaffected (different persistence path).

Commit `9962432` (Jul 10 02:16, session 9) replaced the fallback with
`DbPathResolver.ResolveTradingDbPath(...)` — the correct fix. Session 9 was stall-killed before
it could verify, so the fix sat unproven while sessions 10–11 chased stale theories.
