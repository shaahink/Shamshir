## Root Cause Analysis (read this in full before writing any code)

### RC-1: `BacktestRunner` starts only the CLI — the engine is never started

`src/TradingEngine.CTraderRunner/BacktestRunner.cs:RunAsync()` launches `ctrader-cli` but never starts the engine subprocess. The named pipe (`trading-engine`) has no server. The cBot retries but eventually gives up or hangs. The user must run the engine manually in a separate terminal for any connection to happen.

**Fix:** `BacktestRunner.RunAsync()` must start the engine as a subprocess first, wait for it to be ready (pipe listening), then launch the CLI. Kill the engine subprocess on completion.

### RC-2: Serilog hardcoded to `MinimumLevel.Information` — diagnostic logs invisible

`src/TradingEngine.Host/Program.cs:12`:
```csharp
Log.Logger = new LoggerConfiguration()
    .MinimumLevel.Information()   // ← hardcoded, ignores appsettings.json
```

All `LogDebug` and `LogTrace` calls in `EngineWorker` ("Not enough bars", "No signal", "Evaluating") are silently dropped. We cannot see why no signal fires.

`appsettings.json` already has `"Serilog": { "MinimumLevel": "Debug" }` — it's just never used.

**Fix:** Replace hardcoded config with `ReadFrom.Configuration(builder.Configuration)`.

### RC-3: `DrawdownTracker` hardcoded to $100k initial balance

`src/TradingEngine.Host/Program.cs:59`:
```csharp
var tracker = new DrawdownTracker();
tracker.Initialize(100_000);   // ← hardcoded
```

When the cBot connects with `--balance=200000` (or any other value), the tracker is already initialized with the wrong balance. Drawdown percentages will be wrong from the start.

**Fix:** Do not call `Initialize` at startup. Instead, call `Initialize(update.Balance)` from `HandleAccountUpdate` when the first AccountUpdate arrives (guard with `!_initialized`).

### RC-4: `CalculateLotSize` uses wrong entry price for SL distance

`src/TradingEngine.Risk/RiskManager.cs:86`:
```csharp
var entryPrice = intent.LimitPrice ?? new Price(equity.Equity);
```

For a market order, `intent.LimitPrice == null`, so `entryPrice = new Price(equity.Equity)`. The account equity (e.g. 100000) is being used as an FX price. The SL distance computed from price 100000 to SL 1.085 is ~999,000 pips — a nonsense value. Lot sizing produces a near-zero result and `Math.Max(0, minLots)` saves it to 0.01 lots, but only accidentally.

`OrderDispatcher.DispatchAsync:29` correctly uses `currentMid` for SL distance: `var entryPrice = intent.LimitPrice ?? new Price(currentMid)`.

**Fix:** Add `decimal currentMid` as a parameter to `CalculateLotSize`. Pass it from `OrderDispatcher.DispatchAsync`. Use `intent.LimitPrice ?? new Price(currentMid)` as the entry price for SL distance calculation.

Interface change required: `IRiskManager.CalculateLotSize(TradeIntent, EquitySnapshot, RiskProfile, decimal currentMid)`.

### RC-5: `SubmitOrderAsync` returns a locally-generated GUID — positions never tracked

`src/TradingEngine.Infrastructure/Adapters/NamedPipeBrokerAdapter.cs:88`:
```csharp
public async Task<Guid> SubmitOrderAsync(OrderRequest request, CancellationToken ct)
{
    await SendCommandAsync("SubmitOrder", request, ct);
    return Guid.NewGuid();   // ← random, never matches cBot's position ID
}
```

The cBot executes the order and sends back an `ExecutionEvent` with cTrader's own position ID (an integer, formatted as a Guid string). The engine tracks a completely different GUID. `_positionTracker.TrackOrder(orderCtx.OrderId, ...)` stores an ID that will never match any incoming ExecutionEvent. Positions are never marked as filled, SL/TP hits are never processed.

**Fix:** The `SubmitOrder` command sent to the cBot must include a client-assigned correlation ID. The cBot's `OrderCommandHandler` must echo this ID back in the ExecutionEvent. Use a `Guid` generated by the engine before the command, and include it as `ClientOrderId` in the payload.

### RC-6: Short backtest period — too quiet to generate signals

The smoke test uses `--start=01/01/2024 --end=05/01/2024` (January 1–5). This is a 4-day holiday window with minimal volatility. TrendBreakout needs a 20-bar high/low breakout — unlikely during the New Year quiet period.

Default backtest in the UI should use at least 3 months of data (e.g. 2024-01-15 to 2024-04-15) to ensure enough volatility for signals to fire.

### RC-7 (minor): Bar processor / tick processor race

`ProcessBarsAsync` and `ProcessTicksAsync` are concurrent tasks. A tick for bar N+1 can arrive (and evaluate strategies) before bar N has been stored by `ProcessBarsAsync`. This is a race that affects the first 1–2 ticks after each bar close. Not the root cause of zero trades but affects signal timing correctness.

**Fix (low risk):** At the start of the tick evaluation loop, drain any pending bars from the bar channel before calling `BuildBarSnapshot`.

- `Engine:ActiveStrategyIds=trend-breakout` (keep to single strategy for now)
- `ASPNETCORE_ENVIRONMENT=Development`

Add a timeout: if the pipe doesn't appear within 10 seconds, throw `TimeoutException("Engine failed to start — pipe not ready")`.

Add logging for each step: engine started (PID), pipe ready, CLI started, CLI done, engine killed.

### Phase 5E — Fix position tracking: correlate engine orderId with cBot (RC-5)

This is a two-part fix: engine side and cBot side.

**Engine side — `NamedPipeBrokerAdapter.SubmitOrderAsync`:**

Generate the GUID before sending, include it in the command payload:
```csharp
public async Task<Guid> SubmitOrderAsync(OrderRequest request, CancellationToken ct)
{
    var clientOrderId = Guid.NewGuid();
    await SendCommandAsync("SubmitOrder", new
    {
        ClientOrderId = clientOrderId,
        request.Symbol,
        request.Direction,
        request.Lots,
        request.StopLoss,
        request.TakeProfit,
        request.OrderType,
    }, ct);
    return clientOrderId;
}
```

**cBot side — `OrderCommandHandler.cs` (in `TradingEngine.Adapters.CTrader`):**

When handling `SubmitOrder`:
- Parse `ClientOrderId` from payload
- On position open (in `OnPositionOpened`), match the new position to the last pending `ClientOrderId`
- Send `ExecutionEvent` with `OrderId = ClientOrderId` (not cTrader's int position ID)

The cBot must hold a queue of pending `ClientOrderId` values and pop one when a position opens.

Add a `Queue<Guid> _pendingClientOrderIds = new()` to `TradingEngineCBot`. In `OrderCommandHandler.Handle("SubmitOrder")`, enqueue the ClientOrderId. In `OnPositionOpened`, dequeue and use it in the ExecutionEvent payload.

**Note:** This fix requires a cBot rebuild and re-deploy to cTrader. After the fix, the `.algo` file must be rebuilt: `dotnet build TradingEngine.Adapters.CTrader`.

### Phase 5F — Fix minor bar/tick race (RC-7)

**File:** `src/TradingEngine.Host/EngineWorker.cs`, in `ProcessTicksAsync`, before the strategy loop:

Add a method to drain pending bars:
```csharp
private async Task DrainPendingBarsAsync()
{
    // BarStream is also consumed by ProcessBarsAsync — use the shared dict read instead.
    // Cannot double-consume the channel. Instead, just yield to let ProcessBarsAsync run.
    await Task.Yield();
}
```

Actually the correct fix here: store bars in a separate ConcurrentQueue that both tasks share, OR use a combined event dispatch. For now, add `await Task.Yield()` at the top of each tick iteration to give the bar processor a chance to run. This is not perfect but reduces the race window significantly without restructuring.

```csharp
// In ProcessTicksAsync, inside the tick loop, BEFORE strategy evaluation:
await Task.Yield();  // yield to bar processor
```
